# ข้อความปรับแก้รายงานให้ตรงกับระบบปัจจุบัน

เอกสารนี้เป็นข้อความภาษาไทยสำหรับนำไปแทนหรือปรับในรายงานหลักให้ตรงกับโค้ดในโปรเจกต์นี้

## 1. ภาพรวมระบบ

ระบบ Silent Call เป็นระบบสื่อสารเสียงภายในเครือข่าย LAN ที่ออกแบบให้ผู้ใช้งานสองฝ่ายสามารถติดต่อกันผ่านหมายเลขโทรศัพท์จำลอง 10 หลัก โดยมีเซิร์ฟเวอร์กลางทำหน้าที่เพียงลงทะเบียนผู้ใช้งานและบอกตำแหน่งเครือข่ายของปลายทางเท่านั้น หลังจากนั้นผู้โทรและผู้รับจะเชื่อมต่อกันโดยตรงแบบ peer-to-peer ผ่าน TCP แล้วจึงทำการแลกเปลี่ยนกุญแจ เข้ารหัสข้อมูลเสียง และสื่อสารกันโดยที่เซิร์ฟเวอร์ไม่สามารถถอดรหัสเสียงได้

## 2. ขั้นตอนการทำงานของระบบ

1. ผู้ใช้แต่ละฝ่ายกำหนดหมายเลขโทรศัพท์จำลอง 10 หลัก, IP ของเซิร์ฟเวอร์ และพอร์ตสำหรับรอรับสาย
2. โปรแกรมลงทะเบียนข้อมูลดังกล่าวกับเซิร์ฟเวอร์กลาง
3. เมื่อผู้โทรระบุหมายเลขปลายทาง ระบบจะสอบถามเซิร์ฟเวอร์เพื่อขอ IP และพอร์ตของผู้รับ
4. ผู้โทรเชื่อมต่อโดยตรงไปยังเครื่องผู้รับ
5. ทั้งสองฝ่ายแลกเปลี่ยน Public Key แบบ X25519 เพื่อสร้าง Shared Secret
6. Shared Secret ถูกนำเข้าสู่ HKDF-SHA256 เพื่อสร้าง Session Key สำหรับ AES-256-GCM
7. ก่อนเริ่มส่งเสียง ทั้งสองฝ่ายต้องยืนยันตัวตนผ่านคำถามและคำตอบที่กำหนดร่วมกันเอง และต้องกด Approve ทั้งสองฝั่ง
8. เมื่อยืนยันตัวตนครบทั้งสองฝ่าย ระบบจึงเริ่มส่งข้อมูลเสียงที่เข้ารหัส
9. ระหว่างการสนทนา ระบบจะเปลี่ยนกุญแจทุก 30 วินาที และยอมรับกุญแจเดิมต่ออีก 10 วินาทีเพื่อลดผลกระทบจากแพ็กเก็ตที่มาถึงช้า

## 3. ข้อความที่ควรใช้ในส่วนการพิสูจน์ตัวตน

ระบบปัจจุบันใช้การพิสูจน์ตัวตนแบบ manual verification โดยผู้ใช้งานทั้งสองฝ่ายตกลงคำถามลับร่วมกันเอง ฝ่ายหนึ่งส่งคำถาม อีกฝ่ายตอบคำถาม จากนั้นผู้ใช้แต่ละฝ่ายพิจารณาความถูกต้องของคำตอบและกด Approve Identity ด้วยตนเอง หากยังไม่ได้รับคำตอบหรือยังไม่กดอนุมัติครบทั้งสองฝ่าย ระบบจะยังไม่เปิดการส่งข้อมูลเสียง

หมายเหตุ: ระบบนี้ยังไม่ได้ใช้ Short Authentication String (SAS), Digital Certificate หรือ Public Key Infrastructure (PKI) ดังนั้นรายงานควรอธิบายตามการทำงานจริงข้างต้น และระบุว่าวิธีดังกล่าวเป็นการยืนยันตัวตนเชิงพฤติกรรมของผู้ใช้งาน ไม่ใช่การยืนยันตัวตนด้วยใบรับรองดิจิทัล

## 4. ข้อความที่ควรใช้ในส่วนสถาปัตยกรรม

เซิร์ฟเวอร์กลางของระบบทำหน้าที่เฉพาะด้าน signaling ได้แก่ การลงทะเบียนหมายเลข การจับคู่ผู้ใช้งาน และการตอบกลับข้อมูล IP/Port ของคู่สนทนาเท่านั้น ข้อมูลเสียงและกุญแจเข้ารหัสไม่ได้ถูกประมวลผลโดยเซิร์ฟเวอร์ หลังจากค้นหาปลายทางสำเร็จแล้ว ผู้ใช้ทั้งสองฝ่ายจะเชื่อมต่อกันโดยตรงแบบ peer-to-peer และทำการแลกเปลี่ยนกุญแจกันเอง

## 5. ข้อความที่ควรใช้ในส่วนการเข้ารหัส

ระบบใช้ X25519 สำหรับการแลกเปลี่ยนกุญแจแบบอสมมาตร ใช้ HKDF-SHA256 สำหรับสร้าง Session Key จาก Shared Secret และใช้ AES-256-GCM สำหรับเข้ารหัสข้อมูลเสียงแบบสมมาตรในระดับแพ็กเก็ต การเลือกใช้อัลกอริทึมดังกล่าวช่วยให้ระบบได้ทั้งความลับของข้อมูลและความถูกต้องของข้อมูล พร้อมทั้งมีประสิทธิภาพเพียงพอสำหรับงานเสียงแบบเรียลไทม์ใน LAN

## 6. ข้อความที่ควรใช้ในส่วนขอบเขต

สิ่งที่ระบบครอบคลุม

- การลงทะเบียนหมายเลขโทรศัพท์จำลองกับเซิร์ฟเวอร์
- การค้นหาคู่สนทนาใน LAN
- การเชื่อมต่อแบบ peer-to-peer ระหว่างผู้ใช้สองฝ่าย
- การแลกเปลี่ยนกุญแจด้วย X25519
- การสร้าง Session Key ด้วย HKDF-SHA256
- การเข้ารหัสและถอดรหัสข้อมูลเสียงด้วย AES-256-GCM
- การยืนยันตัวตนก่อนเริ่มสนทนาด้วยคำถามและคำตอบ
- การเปลี่ยนกุญแจอัตโนมัติเป็นระยะระหว่างการสนทนา

สิ่งที่ระบบยังไม่ครอบคลุม

- การยืนยันตัวตนด้วย certificate หรือ SAS
- การป้องกัน DoS ด้วย rate limiting หรือ network isolation
- การป้องกันมัลแวร์ที่เครื่องผู้ใช้
- การรองรับผู้ใช้จำนวนมาก
- การสื่อสารผ่านอินเทอร์เน็ตสาธารณะ

## 7. ข้อความที่ควรใช้ในส่วนภัยคุกคามและข้อจำกัด

แม้ว่าระบบจะเข้ารหัสข้อมูลเสียงแบบ end-to-end ระหว่างผู้ใช้ทั้งสองฝ่าย แต่ระบบยังคงมีข้อจำกัดสำคัญ ได้แก่ การยืนยันตัวตนยังพึ่งพาการตัดสินใจของผู้ใช้, เซิร์ฟเวอร์ยังไม่มีมาตรการ rate limiting หรือ audit log, และหากเครื่องปลายทางติดมัลแวร์ ผู้โจมตียังอาจขโมยข้อมูลเสียงได้ก่อนเข้ารหัสหรือหลังถอดรหัส นอกจากนี้ระบบใช้ TCP สำหรับการส่งเสียงซึ่งเหมาะกับการสาธิตและการพัฒนา แต่ยังไม่เหมาะที่สุดสำหรับงาน VoIP ในเครือข่ายที่มีความหน่วงสูง

## 8. หมายเหตุเกี่ยวกับไฟล์โจมตีใน repository

ใน repository นี้มีไฟล์สำหรับสาธิตภัยคุกคามและการโจมตี เช่น `MITM.py`, `mitm_attack.py` และ `scan_network.py` ไฟล์เหล่านี้ไม่ใช่ส่วนหนึ่งของเส้นทางการทำงานปกติของระบบหลัก จึงควรถูกอธิบายในรายงานว่าเป็นเครื่องมือสำหรับการทดลอง วิเคราะห์ความเสี่ยง หรือสาธิต threat model ไม่ใช่คุณสมบัติของระบบหลัก
