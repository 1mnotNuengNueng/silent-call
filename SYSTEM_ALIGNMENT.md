# Silent Call: เอกสารเทียบข้อกำหนดกับระบบจริงในโปรเจกต์

เอกสารนี้ใช้สำหรับเทียบรายละเอียดใน PDF รายวิชากับระบบที่มีอยู่จริงใน repository นี้ เพื่อช่วยปรับรายงาน สไลด์ และคำอธิบายโครงงานให้ตรงกับสิ่งที่ระบบทำได้จริง

## 1. ระบบปัจจุบันในโปรเจกต์นี้คืออะไร

เส้นทางการทำงานหลักของโปรแกรมมีดังนี้

- `app.py` ใช้เปิดโปรแกรม GUI ด้วย PyQt
- `ui/gui.py` จัดการหน้าจอ การตั้งค่า เสียงเรียกเข้า และสถานะการโทร
- `core/client_core.py` จัดการ signaling lookup, การเชื่อมต่อกับ peer, การแลกกุญแจ X25519, การสร้างกุญแจด้วย HKDF-SHA256, การเข้ารหัสเสียงด้วย AES-256-GCM, ขั้นตอนถาม-ตอบเพื่อยืนยันตัวตน และการเปลี่ยนกุญแจทุก 30 วินาที
- `server.py` เป็น signaling server ทำหน้าที่เก็บ mapping ระหว่างหมายเลขโทรศัพท์จำลองกับ IP/port ของเครื่องผู้ใช้ และตอบกลับข้อมูล peer ที่ต้องการติดต่อ โดยไม่เข้ารหัสเสียงและไม่ถือกุญแจของ session

ลำดับการทำงานจริงของระบบคือ

1. ผู้ใช้แต่ละฝ่ายลงทะเบียนหมายเลขโทรศัพท์จำลอง 10 หลัก และพอร์ตที่ใช้รอรับสายกับ signaling server
2. ฝั่งผู้โทรขอ IP และ port ของปลายทางจาก server
3. ผู้โทรเชื่อมต่อไปยังผู้รับโดยตรงแบบ TCP
4. ทั้งสองฝั่งแลกเปลี่ยน ephemeral public key ด้วย X25519
5. ทั้งสองฝั่งสร้าง AES session key เดียวกันผ่าน HKDF-SHA256
6. ผู้ใช้ทั้งสองฝั่งทำขั้นตอนยืนยันตัวตนด้วยคำถามและคำตอบที่กำหนดกันเอง
7. เมื่อทั้งสองฝ่ายกด approve แล้ว ระบบจึงเริ่มส่งเสียงที่ถูกเข้ารหัส
8. ระบบเปลี่ยนกุญแจทุก 30 วินาที และยอมรับกุญแจเดิมต่ออีก 10 วินาทีเพื่อลดปัญหาแพ็กเก็ตมาช้า

## 2. ส่วนที่สอดคล้องกับ PDF อยู่แล้ว

- เซิร์ฟเวอร์ทำหน้าที่เป็น signaling component ไม่ใช่ media relay
- ข้อมูลเสียงถูกเข้ารหัสแบบ end-to-end ระหว่าง client ทั้งสองฝั่ง
- ใช้ X25519 สำหรับ key exchange
- ใช้ HKDF-SHA256 สำหรับสร้าง symmetric key
- ใช้ AES-256-GCM สำหรับเข้ารหัสข้อมูลเสียง
- ใช้ ephemeral key และมี key rotation
- มีขั้นตอนยืนยันตัวตนก่อนเริ่มส่งเสียง
- ระบบถูกออกแบบเพื่อการใช้งานใน LAN และรองรับการสื่อสารแบบจำกัด

## 3. ส่วนที่ควรแก้ใน PDF ให้ตรงกับ repository นี้

### การยืนยันตัวตน

ใน PDF มีบางช่วงที่อธิบายแนวทางป้องกัน MitM ในระดับที่สูงกว่า implementation จริง เช่น SAS หรือ digital certificate แต่ระบบปัจจุบันยังไม่ได้ทำสองส่วนนั้น

สิ่งที่ระบบทำจริงคือ

- ฝั่งหนึ่งส่งคำถามยืนยันตัวตนแบบเข้ารหัส
- อีกฝั่งตอบคำถามนั้นกลับมา
- ผู้ใช้แต่ละฝ่ายกด approve ด้วยตนเอง
- ระบบจะเริ่มส่งเสียงก็ต่อเมื่อได้รับการ approve ครบทั้งสองฝั่ง

ดังนั้นในรายงานควรอธิบายว่าเป็น **manual shared-secret verification** หรือการยืนยันตัวตนด้วยคำถามลับร่วมกัน ไม่ควรอธิบายว่าเป็น SAS และไม่ควรอ้างว่าเป็น certificate-based authentication

### Signaling และ transport

สถาปัตยกรรมจริงของระบบคือ

- Signaling: client ติดต่อ server ผ่าน TCP เพื่อ register และ lookup peer
- Media: ผู้ใช้สองฝั่งเชื่อมต่อกันโดยตรงแบบ TCP
- เฟรมข้อมูลเสียงและ key exchange ไม่ได้วิ่งผ่าน `server.py`

ถ้าใน PDF เขียนราวกับว่า server เป็นตัวส่งต่อ public key หรือส่งต่อเสียง ควรปรับข้อความส่วนนั้น

### การอ้างมาตรการป้องกันภัยคุกคาม

ในโค้ดปัจจุบัน **ยังไม่มี implementation จริง** สำหรับ

- certificate pinning
- PKI
- MFA
- malware protection
- rate limiting สำหรับ DoS
- VLAN isolation
- secure enclave หรือ hardened memory

สิ่งเหล่านี้ควรเขียนเป็นแนวทางลดความเสี่ยงในอนาคต หรือข้อเสนอแนะเพิ่มเติม ไม่ควรเขียนว่าเป็นฟีเจอร์ที่ระบบปัจจุบันทำได้แล้ว

### ประเด็นเรื่องการบันทึกเสียง

เส้นทางทำงานหลักของระบบไม่ได้บันทึกเสียงสนทนา แต่ใน repository นี้มีไฟล์สำหรับสาธิตการโจมตีและการทดลอง ได้แก่

- `MITM.py`
- `mitm_attack.py`
- `scan_network.py`

ไฟล์เหล่านี้ไม่ใช่ส่วนหนึ่งของระบบหลัก จึงควรแยกไปอธิบายในหัวข้อ threat demonstration หรือการประเมินความเสี่ยง ไม่ควรนับเป็นฟังก์ชันปกติของระบบ

## 4. ข้อจำกัดสำคัญที่ควรระบุในรายงาน

- salt ที่ใช้ในกระบวนการ derive key ยังถูกกำหนดคงที่อยู่ในโค้ด
- การยืนยันตัวตนยังขึ้นกับพฤติกรรมของผู้ใช้ จึงอ่อนกว่าระบบ authenticated key exchange จริง
- signaling server ยังเป็นระบบขนาดเล็กและไม่มี rate limiting, audit log หรือ session management ที่เข้มงวด
- ระบบถูกออกแบบเพื่อ demo ใน LAN ขนาดเล็ก ไม่ใช่ระบบ production scale
- ระบบใช้ TCP สำหรับ media ซึ่งทำให้พัฒนาและเดโมง่าย แต่ไม่ใช่ตัวเลือกที่เหมาะที่สุดสำหรับงาน voice real-time ในเครือข่ายที่มีปัญหาความหน่วง
- หากเครื่องปลายทางติดมัลแวร์ ผู้โจมตียังสามารถขโมยเสียงก่อนเข้ารหัสหรือหลังถอดรหัสได้

## 5. สิ่งที่เพิ่มหรือแก้ให้ระบบแล้วในรอบนี้

- หน้า Settings สามารถบันทึกค่ากลับไปที่ `app_config.json` ได้จริง
- หลังบันทึกค่าใหม่ ระบบจะ reconnect ด้วยค่าชุดใหม่ทันที
- ข้อความที่เพี้ยนใน flow การยืนยันตัวตนถูกปรับให้อ่านเข้าใจได้ชัดเจนขึ้น เพื่อให้สาธิตระบบได้ง่ายและไม่สับสน

## 6. ข้อความแนะนำที่ใช้แทนในรายงานได้

สามารถใช้ข้อความลักษณะนี้ในรายงาน:

"ระบบใช้เซิร์ฟเวอร์กลางเพื่อการลงทะเบียนและค้นหาคู่สนทนาเท่านั้น หลังจากค้นหาปลายทางสำเร็จ อุปกรณ์ทั้งสองฝ่ายจะเชื่อมต่อกันโดยตรงผ่าน TCP จากนั้นทำการแลกเปลี่ยนกุญแจแบบ X25519 สร้าง Session Key ผ่าน HKDF-SHA256 ยืนยันตัวตนด้วยคำถามลับและการอนุมัติจากผู้ใช้ทั้งสองฝ่าย แล้วจึงเริ่มส่งข้อมูลเสียงที่ถูกเข้ารหัสด้วย AES-256-GCM โดยเซิร์ฟเวอร์ไม่สามารถเข้าถึงกุญแจ session และไม่สามารถถอดรหัสข้อมูลเสียงได้"

## 7. การแยกไฟล์ core ออกจากไฟล์ทดลอง

ไฟล์ที่เป็นระบบหลัก

- `app.py`
- `ui/gui.py`
- `ui/pages/*.py`
- `core/client_core.py`
- `server.py`
- `app_config.json`

ไฟล์ที่เป็น legacy, experiment หรือ attack-support

- `client.py`
- `MITM.py`
- `mitm_attack.py`
- `scan_network.py`
- `generate_key.py`

เวลาแก้รายงาน ควรอธิบายระบบหลักก่อน แล้วค่อยอ้างไฟล์กลุ่มโจมตีในส่วนการทดลองโจมตีหรือการวิเคราะห์ภัยคุกคาม
